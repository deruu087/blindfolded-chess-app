# 3.52 EUR Origin Analysis

## Summary
The **3.52 EUR** value is a hardcoded test/mock amount that appears in multiple places in the codebase. It's being actively rejected throughout the application to prevent test data from appearing in production.

---

## Where 3.52 EUR Comes From

### 1. **payment-success.html** (Line 168) - PRIMARY SOURCE
```javascript
// Use plan-based amounts if not in URL (these are the actual Dodo Payments prices)
let finalAmount = amount;
if (!finalAmount) {
    finalAmount = planType === 'monthly' ? '3.52' : '8.90';
    console.log('💰 Using plan-based amount:', finalAmount, currency);
}
```

**Context:** This is a **fallback value** used when the payment amount is not provided in the URL parameters. The comment claims these are "actual Dodo Payments prices" but this appears to be outdated or incorrect.

**Issue:** If a user completes payment but the amount isn't in the URL, this hardcoded 3.52 EUR gets saved to the database instead of the real amount.

---

### 2. **Test Scripts** - Used for Testing
- `simple-webhook-test.js` (Line 10): Default test amount
- `sync-existing-subscription.js` (Line 10): Default test amount  
- `test-webhook.json` (Line 7): Test webhook payload
- `test-webhook.sh` (Line 19): Test script
- `LOCAL-WEBHOOK-TESTING.md`: Documentation example

**Purpose:** These are legitimate test values used for development and testing.

---

### 3. **Actual Subscription Prices (Displayed)**
From `subscription.html`:
- **Monthly:** $4.99 USD
- **Quarterly (3 months):** $12.99 USD

**Note:** The displayed prices are in **USD**, not EUR. The 3.52 EUR value doesn't match:
- $4.99 USD ≈ 4.52 EUR (at current exchange rates)
- 3.52 EUR is lower than the actual price

---

## Why 3.52 is Rejected

The codebase has **extensive rejection logic** to prevent 3.52 from appearing in production:

### Rejection Points:

1. **supabase-helpers.js** - `getUserSubscription()` (Lines 568, 590)
   - Rejects subscriptions with `amount_paid == 3.52`
   - Logs: "REJECTING SUBSCRIPTION WITH 3.52 - THIS IS HARDCODED TEST DATA"

2. **supabase-helpers.js** - `getPaymentHistory()` (Line 817)
   - Filters out payments with `amount == 3.52`
   - Logs: "Rejecting payment with amount 3.52 (mock data)"

3. **profile.html** - Multiple rejection checks (Lines 1612, 1653, 1660, 1762)
   - 4 different checkpoints that reject 3.52 subscriptions
   - 1 checkpoint that filters 3.52 payments

4. **games.html** - (Newly added)
   - Will display subscription data, but 3.52 subscriptions are already rejected upstream

---

## The Problem

1. **Outdated Fallback:** `payment-success.html` uses 3.52 as a fallback, which can save incorrect amounts to the database.

2. **Currency Mismatch:** Displayed prices are in USD ($4.99), but the fallback is in EUR (3.52).

3. **Incorrect Amount:** 3.52 EUR doesn't match the actual $4.99 USD price (should be ~4.52 EUR).

4. **Test Data Leakage:** If the webhook doesn't fire or URL parameters are missing, test data (3.52) could be saved instead of real payment data.

---

## Recommended Fixes

### Option 1: Remove the Fallback (Recommended)
Remove the hardcoded fallback in `payment-success.html` and require the amount to come from:
- URL parameters (from Dodo Payments redirect)
- Webhook data (from Dodo Payments webhook)
- Don't save subscription if amount is missing

### Option 2: Use Correct Prices
If a fallback is needed, use the correct EUR-converted prices:
- Monthly: ~4.52 EUR (from $4.99 USD)
- Quarterly: ~11.70 EUR (from $12.99 USD)

### Option 3: Fetch from Dodo Payments API
Query Dodo Payments API to get the actual product price instead of hardcoding.

## Conclusion

**3.52 EUR** is a **hardcoded test/mock value** that:
1. Originates from `payment-success.html` as a fallback when amount is missing
2. Doesn't match the actual subscription prices ($4.99 USD / $12.99 USD)
3. Is actively rejected throughout the codebase to prevent test data leakage
4. Should be removed or replaced with correct values/API calls

The extensive rejection logic shows this was a known issue that was addressed by filtering, but the root cause (hardcoded fallback in payment-success.html) should still be fixed.
